Screen Agent MVP Architecture

Living architecture notes for the local 5090 screen-recording intelligence MVP. The current build turns video into sampled frames, OCR/UI observations, timeline events, evidence clips, a bug-report-style Markdown report, and a small local question-answering layer.

Phase: Week 1 core plus Week 2 demo UX pieces Runtime: Ubuntu/WSL2 on RTX 5090 Project: screen-agent-mvp

What Exists Now

Built Local analysis pipeline

A CLI and Gradio app can ingest a screen recording, sample frames, run model adapters, and write artifacts.

Built Strict 5090 path

The WSL copy has CUDA-visible PyTorch, PaddleOCR, OmniParser wrapper support, and Qwen2.5-VL reasoning.

Built Fallback path

Deterministic fallback OCR, UI parsing, and report generation keep tests and demos runnable without model weights.

End-to-End Pipeline

Video Input
Local .mp4 or .webm uploaded through CLI or Gradio.
Probe + Sample
OpenCV extracts metadata, interval frames, and scene-change frames.
Keyframes
JPEG frames are saved under each run's keyframes directory.
OCR
PaddleOCR in strict mode, fallback text-region detector otherwise.
UI Parsing
OmniParser wrapper in strict mode, heuristic contour parser otherwise.
Frame Observations
Timestamped OCR blocks, UI elements, frame paths, and warnings.
Timeline + Reasoning
Events are derived locally; Qwen2.5-VL or template engine builds report.
Artifacts + Ask
Markdown, JSON, evidence clips, gallery frames, and local QA over the run.
Current output bundle
report.md, timeline.json, summary.json, keyframes/*.jpg, and evidence_clips/*.mp4.

Runtime Architecture

User-facing entry points

  • screen-agent diagnostics: checks system/runtime basics.
  • screen-agent models-check --strict: verifies required model stack.
  • screen-agent make-demo-video: creates a synthetic checkout-failure recording.
  • screen-agent analyze: runs the full video analysis pipeline.
  • screen-agent ask: answers questions over an existing run.
  • screen-agent-web: launches the Gradio upload, timeline, gallery, and QA UI.

Execution environments

  • Windows workspace: source of truth for project files and copied artifacts.
  • WSL Linux filesystem: preferred strict-model runtime to avoid slow or fragile /mnt/c virtualenv behavior.
  • RTX 5090: used by PyTorch/Qwen2.5-VL in strict mode; PaddleOCR is currently configured on CPU.
  • OmniParser repo: external local checkout invoked through SCREEN_AGENT_OMNIPARSER_COMMAND.

Model Stack

Layer Strict-mode implementation Fallback implementation Why it exists
OCR PaddleOcrEngine using PaddleOCR FallbackOcrEngine estimates text-like regions Reads visible app text, errors, labels, and button copy from frames.
UI parser OmniParserUiEngine via local JSON command wrapper HeuristicUiEngine using OpenCV contours Finds controls, fields, buttons, and other UI objects from pixels.
Reasoning QwenVlReasoningEngine with Qwen/Qwen2.5-VL-7B-Instruct TemplateReasoningEngine Turns screenshots plus structured context into a bug report with evidence.
Video processing OpenCV plus Pillow Same Probes videos, samples frames, saves keyframes, exports short evidence clips.

Current known caveat: OmniParser detection works through the wrapper, while Florence icon captions are disabled by default because that path was not stable under the current Transformers setup.

Detailed Flow

  1. The user uploads or passes a local recording path through CLI or Gradio.
  2. AnalysisConfig resolves preset defaults, model mode, sample rate, frame limit, VLM model ID, and OmniParser command.
  3. sample_video_frames probes metadata, selects interval timestamps, adds scene-change timestamps, and writes JPEG keyframes.
  4. build_engines chooses strict model adapters or deterministic fallbacks.
  5. Each sampled frame becomes a FrameObservation after OCR and UI parsing.
  6. build_timeline_events flags initial state, scene changes, possible error states, and final state.
  7. The reasoning engine receives selected key observations and builds a structured Report.
  8. Evidence clips are exported around report evidence timestamps when OpenCV can write the clip.
  9. write_report_bundle writes the run artifacts.
  10. answer_run_question can later answer questions from stored JSON without rerunning models.

Current Product Shape

Demo app

The Gradio UI is the first app-like experience. It supports video upload, preset selection, sampling controls, strict-model mode, report rendering, a timeline table, keyframe gallery, artifact paths, and a question box for completed runs.

Developer workflow

The CLI is the reliable local workflow for debugging. It can generate the synthetic demo video, run fallback or strict model analysis, and inspect a completed run with natural-language questions.

Product thesis: this is a local prototype of an agent-ready video intelligence API. The current artifacts already look like an API response: a run object, frame observations, timeline events, report evidence, and queryable state.

Agentic UX Angles

Browser-agent evaluator

Judge whether an AI browser agent completed a task from pixels, not just logs.

Bug-report copilot

Convert user-submitted screen recordings into reproducible issue reports with evidence.

Support-session summarizer

Summarize long support recordings into the actual user-visible failure and timestamped clips.

QA run inspector

Turn smoke-test recordings into pass, fail, or uncertain verdicts with visible evidence.

Replay search

Ask where an error, field, modal, or checkout state appears in archived footage.

Action trace grounding

Pair future click/URL telemetry with visual state to explain what happened and why.

Roadmap

Week 1
Basic local demo

CLI, Gradio app, video sampling, keyframes, OCR, UI parsing, VLM report, Markdown and JSON artifacts. This is substantially built.

Week 2
Better demo UX

Timeline events, evidence clips, presets, and run-level QA are implemented. The remaining work is mostly polish and validation on more videos.

Week 3
Browser recorder and agent evaluation

Add Playwright/local browser recording, optional telemetry, verdicts, and before/after comparison.

Week 4
Local developer API

Expose POST /runs, run status, timeline, report, ask, and evidence endpoints.

Week 5
Evaluation gate

Test failed checkout, signup/login, onboarding, support recordings, browser-agent attempts, successful tasks, and noisy sessions.

Future API Shape

Run lifecycle

  • POST /runs: upload video and start analysis.
  • GET /runs/{run_id}: fetch metadata, status, and summary.
  • GET /runs/{run_id}/timeline: fetch timestamped events and frame observations.

Developer outputs

  • POST /runs/{run_id}/ask: ask questions over the stored run.
  • GET /runs/{run_id}/report.md: retrieve the rendered bug report.
  • GET /runs/{run_id}/evidence/{id}: retrieve evidence frames or clips.
